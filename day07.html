why this matters · the road to honest applicability-domain flags
Today's /similar is the structural input that Day 17 turns into honest in_domain.

On Day 5, the /predict response included in_domain: true for every call — a placeholder, because no applicability-domain methodology was implemented yet. Today's similarity service makes that honest. By the end of Day 17, every /predict call will internally call /similar(top_k=10) and compute:

in_domain := mean(cosine similarities of top-10 neighbors) ≥ 0.45

The 0.45 threshold is a placeholder until Day 17 tunes it against a held-out test set of chemotype-novel compounds. The mean-of-top-10 rather than max-of-top-1 is robust to single false-positive matches in the training corpus. The contract is regulator-friendly: every prediction either declares its applicability or admits it cannot. No more silent extrapolation.

Two halves of the same pipeline. Build-time runs once today — embed all 23K curated compounds, write the parquet, build the HNSW index, register a dataset_registry row. V1 plan rebuilds nightly via cron; for POC we accept that the index is a snapshot. Query-time runs per request — canonicalize, embed (32 ms is the bottleneck), HNSW search (2 ms in a 23K-vector index), enrich + serialize. End-to-end p95 budget is 45 ms; the alert at 50 ms gives 10% headroom over the budget.

Worth noting what is not in this diagram. No deduplication across the 4 source datasets (Tox21 + ClinTox + hERG + DILIrank have some compound overlap; V1 handles canonical_id dedup). No multiple embedding spaces (V1 may add Morgan-FP-based similarity alongside ChemBERTa-CLS for ensemble retrieval). No filtering by endpoint (V1 may filter "show me only hERG-positive neighbors" via a metadata WHERE clause). Today is the simple shape: one embedding space, one index, one cosine top-k.

DR-7.1 retired

ChemBERTa OOM on the 23K-compound batch loop.

Predicted Per-call ChemBERTa CPU forward is ~30 ms and ~150 MB working set; naively looping 23K compounds in one worker process risks resident memory climbing as tokenizer + intermediates accumulate. Risk: OOM kill mid-batch, partial parquet, restart from scratch. Mitigation: chunk into 8 sub-tasks via Celery group (each 2,882 compounds, fresh tokenizer per chunk); concat step combines partials.
What happened Mitigation worked exactly as designed. 8 chunk tasks dispatched 11:04:08; each ran 30-43 sec wall-clock; all 8 complete by 11:05:34; concat callback complete at 11:05:58. Total wall-clock 1m50s — 8-way parallel beat the standup serial estimate (11m26s) by 84%. Zero OOMs; zero retries; peak per-worker resident memory was 1.3 GB (well under any reasonable ceiling). The "mitigation cost" was negative — structuring as a chord made the work both safer and faster.
Disposition Retired — chord pattern committed for embed_dataset; same pattern carries forward to Day-12 batch enrichment (which will fan out predictions + similarity over user-submitted batches). Pattern documented in PR-#23 commit message as "8-way Celery chord for embarrassingly-parallel batch work over the curated corpus."
DR-7.2 mitigated

ChemBERTa cold-start on v0.3.0 may exceed user-tolerable latency.

Predicted v0.2.0 ships only joblib XGBoost (10-50 MB, cold-load <200 ms); v0.3.0 adds transformers + DGL paths where ChemBERTa cold-load involves downloading from MinIO (~330 MB checkpoint + tokenizer dir) and instantiating a 12-layer transformer. What happened First /predict to NR-AhR after the v0.3.0 deploy took 9.4 sec wall-clock — under the 12 sec budget but clearly user-visible. Subsequent warm calls 87-114 ms (mean 93 ms). Cold/warm ratio ~100x. Day-11 pre-warm hook remains valid but not emergency; the production /predict path works today for users who tolerate a one-time cold-load.
Disposition Mitigated — under-budget cold-start preserves the production path's usability for v0; Day-11 work hardens it. The pattern "first-request-of-the-day pays the cold-start tax" is acceptable for an internal POC; V1 productionization needs the pre-warm hook regardless.
