Morgan FP
radius 2 · 2,048 bits · uint8

Circular substructure fingerprint. Every "1" bit means a particular 2-hop atom-environment is present in the molecule. Captures local connectivity — the input that XGBoost and random forest baselines use directly on Day 5.

RDKit descriptors
208 scalars · float32

Numerical molecular properties — logP, TPSA, molecular weight, ring counts, rotatable bonds, Bertz complexity, Lipinski violations, etc. Computed via RDKit's Descriptors._descList. Captures global properties — complementary to fingerprints, useful as a calibrated baseline alongside Morgan.

Per-row provenance
2 metadata cols + sidecar JSON

Each compound row carries its canonical SMILES + scaffold-split assignment. The sidecar JSON has input/output sha256, RDKit version, curate.py + featurize.py git-hashes, rejection counts. This is what gets a regulator past "where did these features come from" — full chain-of-custody.

The diagram is the day's contract. By 17:30 IST every arrow has fired at least once: 4 raw dataset_registry rows triggered 4 curate_dataset tasks via Redis, 8 worker processes consumed in parallel, MinIO has 4 fresh curated parquets and 4 fresh feature parquets with sidecar JSONs, the registry has 8 new rows tagged v0-prod-curated / v0-prod-features, and Grafana shows the whole thing on a Celery dashboard. The output of today's loop is the input of tomorrow's: Day 5 baselines query dataset_registry WHERE version='v0-prod-features' and pick up exactly those parquets.

Worth noting what is not in this diagram. No model training (Day 5). No HTTP API in front of the workers (Day 11 orchestrator). No GPU acceleration (V1 Horizon-01). Today is just the data plumbing actually works at platform scale — the proof that putting work on Celery, watching it complete, and shipping the outputs back through the same persistence tier is a transaction the platform can perform reliably. Every subsequent day extends this loop with one more capability.

Disposition Retired — smoke-task discipline becomes the standard pre-flight for any future async-layer addition (Day 6 GNN training, Day 11 orchestrator, Day 12 batch jobs). Pattern committed in docs/runbooks/celery-bringup.md.
DR-4.2 mitigated

Dockerfile drift between core-api and tox-inference — copy-paste leading to silent divergence.

Predicted Two service Dockerfiles inevitably diverge over time. One installs libgomp1, the other forgets; one pins Python 3.12.7, the other floats to 3.12.x. The drift is invisible until a Day-N runtime misbehavior traces back to a missing library on one of the two images.
What happened Extracted aitip-base.Dockerfile with the digest-pinned Python base, all shared system deps, non-root user, tini, env-var defaults. Both core-api.Dockerfile and tox-inference.Dockerfile FROM aitip-base:0.1.0. Drift surface area reduced to the dataset-of-deltas (PyTorch CPU + sklearn for tox-inference; FastAPI + boto3 for core-api). Future system-dep additions land in the base; the service files don't change.
Disposition Mitigated structurally — the drift surface is now the smallest it can be. Trivy 0/0 across all three images confirms no leaks from base to services. Registry pushes operational; Day 5 deploys via docker run.
DR-4.3 retired

Flower scoped-key drift — using the wrong Vault credential for the monitoring UI.

Predicted Day 3 generated three scoped credentials (readonly, readwrite, backup-user). Easy to give Flower the wrong one (e.g., the broker-only key when it actually needs read access to the result backend keys too). Mitigation: use the same AITIP_REDIS_URL Sarthak's workers use, single source of truth.
What happened Flower started under the same supervisord environment file as the workers; loaded the same AITIP_REDIS_* env-vars; used the same Redis password. No scoped-key drift, no separate credential to misplace. Flower UI showed 8 workers in "Online" tab within 11 seconds of supervisorctl start.
Disposition Retired — single-source-of-truth pattern works. Same approach applies to Day-5 model serving (FastAPI core-api + Celery client share env-file) and Day-9 RAG (vLLM + ChromaDB client share token-via-Vault).

Pattern noted. Two of the three risks today were closed by discipline-by-default rather than discipline-as-reaction — the smoke task and the env-file pattern are now build-time defaults for any future async or service component.
